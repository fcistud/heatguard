The solution

A site-level scheduler with a verification layer

HeatGuard is a WBGT-driven work-rest-hydration scheduler deployed to the supervisor, not to each worker. It senses, schedules, signals, and verifies — plus a graded acclimatization ramp for new arrivals, the deadliest group and the cheapest to protect.

The four things it does #

📡

1 · Senses

Takes one on-site WBGT reading (a ~$300 meter) or estimates WBGT from weather, per ISO 7243.

📅

2 · Schedules

Outputs the actual work-rest cycle (ACGIH / ISO 7243) and hydration target (ISO 7933 PHS) for the current conditions, work intensity, and worker.

📢

3 · Signals

Drives one cheap, universally understood broadcast — a site horn, light, or one supervisor phone.

🔐

4 · Verifies

Logs every reading, break, drink prompt, and water attestation into a tamper-evident, hash-chained record.

WORK REST IN SHADE DRINK NOW STOP

The verification record doubles as the employer's compliance shield against inspection and fines: cryptographic proof that conditions were monitored and breaks and water were provided. Plus the acclimatization ramp — pure scheduling, zero cost — that protects the new arrivals who actually die.

Why site-level, not per-worker (for now) #

Per-worker hardware is where adoption friction lives — cost, charging, breakage, and surveillance concerns. The adoptable form is one sensor per site, riding on existing infrastructure, with near-zero marginal cost per worker, that gives the employer a productivity story and a liability shield rather than a cost — and supplies exactly the implementation-fidelity layer the evidence says is the real missing piece.

The adoptable wedge

One sensor per site, near-zero marginal cost per worker, is what actually gets adopted. Per-worker wearables are a powerful next step for personalisation and enforcement — see the roadmap — but the site-level core is the wedge that gets in the door. (A per-worker NFC wristband at the water station only, to verify hydration, is an optional premium layer.)

How the pieces fit together #

HeatGuard is one shared, pure, deterministic Python engine. The engine does no I/O — weather ingestion and exports live at the edges — so the numbers are computed in exactly one place and the test suite is trustworthy. A single decision for one (worker, hour) flows through frozen, immutable dataclasses: 

Weather  ──estimate_wbgt──▶  Conditions  ──scheduler.decide──▶  Advisory  ──compliance.append──▶  LogRecord
(tdb, rh,                    (+ wbgt_c,                          (signal, cycle,                   (hash-chained)
 wind, solar,                 wbgt_source,                        hydration, rationale,
 pressure)                    globe_c,                            risk_score, ...)
                              met_category)

Every WBGT value carries its provenance (liljegren | fallback | measured) all the way into the compliance log and the UI — the approximation is never hidden. The full breakdown of each scientific step is on the How it works page.

Five interfaces over one engine #

Every interface is a thin presentation layer over the same engine:

  • CLI — heatguard demo dubai|riyadh (the narrative), roi, backtest, decide, fetch / fetch-demo, sites.
  • FastAPI (uvicorn heatguard.api:app) — /demo/{site}, /timeline/{site}/{date}, /hour/{site}/{date}/{hour} (per-worker recompute, optional ?measured_wbgt=), /impact, /economics, /scale, /backtest, /compliance/{site}/export, POST /decide. Interactive docs at /docs.
  • React dashboard — the primary pitch UI: live signal, the WBGT gauge with an Estimated ⟷ Measured toggle, the calendar-ban-vs-HeatGuard timeline with per-worker controls, the acclimatization tracker, season impact, the business-case panel, and the worker-protection record.
  • Streamlit — the same story, pure-Python, no build step.
  • Notebook — the narrated validation with charts.
